# 002 — PRD: Wspólniak MVP

**Status:** Draft — ready for `/carve`
**Data:** 2026-04-09
**Źródło:** [001-discovery.md](./001-discovery.md)

## Problem Statement

Rodziny chcą dzielić się zdjęciami w prywatnym gronie, ale istniejące opcje mają wady:
- **Facebook/Instagram** — publiczne profile (nawet "private" wycieka), algorytmiczny feed, reklamy, tracking, obce osoby w komentarzach, ryzyko moderacji zewnętrznej.
- **WhatsApp/Telegram groups** — chaos chronologiczny, zdjęcia giną w rozmowach, brak "wall", brak komentarzy pod konkretnym zdjęciem, niska jakość (kompresja).
- **Google Photos albumy współdzielone** — wymagają konta Google każdego członka rodziny, babcia nie ogarnie.
- **iCloud Shared Albums** — tylko Apple, wyklucza Android w rodzinie.
- **Self-hosted Nextcloud/Immich** — overkill, skomplikowany setup, brak UX "social feed", wymagają serwera.

Starsze osoby w rodzinie (babcie, dziadkowie) mają szczególny problem: nie radzą sobie z hasłami, rejestracjami, captch, weryfikacjami email. Każda bariera = nieużywany serwis.

Autor chce uruchomić taki serwis dla swojej rodziny pod `wspolniak.com` i udostępnić kod jako open-source, by inne rodziny mogły zrobić to samo u siebie.

## Solution

**Wspólniak** — prywatny, rodzinny serwis do dzielenia się zdjęciami, zbudowany jako PWA, self-hostowany na Cloudflare.

**Kluczowe różnice wobec istniejących rozwiązań:**

1. **Zero haseł, zero rejestracji.** Admin (głowa rodziny) tworzy konta dla członków i dystrybuuje unikalne magic linki poza systemem. Kliknięcie linku = zalogowanie na zawsze (long-lived cookie). Babcia po prostu klika link, instaluje PWA na telefonie i używa.

2. **Jedna rodzina = jedna instancja.** Zero multi-tenancy. Każda rodzina hostuje własny deployment na Cloudflare przez "Deploy to Cloudflare" button. Dane są fizycznie odseparowane. Zero zależności od scentralizowanej usługi.

3. **PWA z push notifications.** Członkowie rodziny dostają natychmiastowe powiadomienia o nowych zdjęciach i komentarzach, nawet na iOS (po "Dodaj do ekranu głównego").

4. **Uproszczony feed.** Jeden wspólny chronologiczny wall z postami (opis + 1..N zdjęć) i komentarzami. Żadnych algorytmów, żadnych reklam, żadnych lajków — tylko to co potrzebne.

5. **HEIC just works.** iPhone uploaduje HEIC, Cloudflare Images konwertuje do JPEG/WebP w locie, babcia nie musi wiedzieć co to format.

6. **Open-source (AGPL-3.0).** Każdy może uruchomić, zajrzeć w kod, zaufać.

## User Stories

### Admin — zarządzanie instancją

1. Jako nowy self-hoster, chcę wdrożyć Wspólniaka jednym kliknięciem "Deploy to Cloudflare", abym nie musiał konfigurować CLI i secrets.
2. Jako nowy self-hoster, po pierwszym deploy chcę wejść na `/setup` i jednorazowo utworzyć konto admina (nazwa rodziny + moje imię), aby stać się adminem swojej instancji.
3. Jako admin, chcę aby endpoint `/setup` przestał działać po utworzeniu instancji, aby nikt inny nie mógł "zabrać" mojej instancji.
4. Jako admin, chcę utworzyć konto dla członka rodziny podając jego imię, aby wygenerować dla niego unikalny magic link.
5. Jako admin, chcę skopiować magic link członka rodziny, aby wysłać mu go przez SMS/WhatsApp/email.
6. Jako admin, chcę wygenerować nowy link dla członka (revoke + new), jeśli zgubił telefon lub link wyciekł, aby stary link przestał działać.
7. Jako admin, chcę usunąć członka rodziny, aby jego konto przestało działać (soft delete, posty/komentarze zostają jako "usunięty użytkownik").
8. Jako admin, chcę usunąć dowolny post lub komentarz, aby moderować treści (np. ktoś omyłkowo wrzucił coś niewłaściwego).
9. Jako admin, chcę zobaczyć listę wszystkich członków rodziny z ich linkami, aby śledzić kto ma dostęp.

### Członek rodziny — logowanie

10. Jako członek rodziny, chcę kliknąć otrzymany magic link, aby automatycznie zalogować się bez podawania hasła.
11. Jako członek rodziny, chcę aby po pierwszym kliknięciu linku sesja trwała "na zawsze" (długoterminowy cookie), abym nie musiał klikać linku przy każdej wizycie.
12. Jako członek rodziny korzystający z iPhone'a, chcę zainstalować Wspólniak jako PWA z ekranu głównego, aby działał jak natywna aplikacja.
13. Jako członek rodziny korzystający z Androida, chcę zainstalować Wspólniak jako PWA, aby działał jak natywna aplikacja.
14. Jako członek rodziny, chcę zobaczyć jasny komunikat "Dodaj do ekranu głównego" przed włączeniem powiadomień na iOS, aby wiedzieć że to warunek działania push.

### Członek rodziny — tworzenie postów

15. Jako członek rodziny, chcę utworzyć post z opcjonalnym opisem i 1-10 zdjęciami, aby podzielić się wydarzeniem z rodziną.
16. Jako członek rodziny, chcę uploadować zdjęcia w formacie JPEG, PNG, WebP, HEIC lub HEIF, aby nie martwić się formatem mojego telefonu.
17. Jako członek rodziny z iPhone'em, chcę aby moje zdjęcia HEIC były automatycznie konwertowane na format widoczny dla wszystkich, aby babcia na Androidzie też je zobaczyła.
18. Jako członek rodziny, chcę zobaczyć preview moich zdjęć przed publikacją, aby sprawdzić co wrzucam.
19. Jako członek rodziny, chcę zobaczyć progress uploadu dużych plików, aby wiedzieć że coś się dzieje.
20. Jako członek rodziny, chcę otrzymać jasny komunikat błędu jeśli przekroczę limit (>10 zdjęć, >15 MB, >50 postów dzisiaj), aby wiedzieć co poprawić.
21. Jako członek rodziny, chcę edytować opis swojego posta po publikacji, aby poprawić literówkę.
22. Jako członek rodziny, chcę usunąć swój post, aby wycofać coś co nie powinno się tam znaleźć.

### Członek rodziny — przeglądanie feedu

23. Jako członek rodziny, chcę zobaczyć chronologiczny feed wszystkich postów rodziny, aby być na bieżąco.
24. Jako członek rodziny, chcę scrollować feed w nieskończoność (infinite scroll, 20 postów/page), aby łatwo dotrzeć do starszych treści.
25. Jako członek rodziny, chcę zobaczyć miniatury zdjęć ładowane szybko, aby feed nie był wolny.
26. Jako członek rodziny, chcę kliknąć miniaturę i zobaczyć pełne zdjęcie w wysokiej jakości, aby zobaczyć szczegóły.
27. Jako członek rodziny, chcę zobaczyć kto jest autorem posta, kiedy został utworzony, i jego opis.
28. Jako członek rodziny, chcę przeglądać feed offline (ostatnio załadowane posty), gdy straciłem internet, aby nie stracić kontekstu.

### Członek rodziny — komentarze

29. Jako członek rodziny, chcę dodać komentarz pod dowolnym postem, aby zareagować.
30. Jako członek rodziny, chcę zobaczyć wszystkie komentarze pod postem chronologicznie (flat, bez wątków), aby łatwo nadążać.
31. Jako członek rodziny, chcę edytować swój komentarz, aby poprawić literówkę.
32. Jako członek rodziny, chcę usunąć swój komentarz, aby wycofać to co napisałem.

### Powiadomienia push

33. Jako członek rodziny, chcę otrzymać powiadomienie push gdy ktoś doda nowy post, aby dowiedzieć się od razu.
34. Jako autor posta, chcę otrzymać powiadomienie push gdy ktoś skomentuje mój post, aby dowiedzieć się o reakcji.
35. Jako członek rodziny, **nie** chcę otrzymywać powiadomień o moich własnych akcjach, aby nie spamować siebie.
36. Jako członek rodziny, chcę kliknąć powiadomienie i otworzyć Wspólniaka bezpośrednio na tym poście/komentarzu, aby szybko zareagować.
37. Jako członek rodziny, chcę móc wyłączyć powiadomienia w ustawieniach przeglądarki/systemu (standardowy mechanizm), aby mieć kontrolę.
38. Jako system, chcę automatycznie usunąć wygasłe subskrypcje push (HTTP 410 Gone), aby nie marnować zasobów.

### Publiczny landing

39. Jako potencjalny self-hoster, chcę wejść na `wspolniak.com` i zobaczyć publiczny landing page opisujący projekt, aby zrozumieć co to jest.
40. Jako potencjalny self-hoster, chcę zobaczyć link do repozytorium GitHub na landing page, aby zobaczyć kod.
41. Jako potencjalny self-hoster, chcę zobaczyć "Deploy to Cloudflare" button na landing page, aby jednym kliknięciem wdrożyć u siebie.
42. Jako członek rodziny z magic linkiem, nie chcę widzieć landing page'a — chcę być od razu przekierowany do aplikacji po zalogowaniu.

### Obsługa błędów i edge cases

43. Jako członek rodziny z nieprawidłowym/zrevoke-owanym tokenem, chcę zobaczyć jasny komunikat "Link nieaktywny, poproś admina o nowy", aby wiedzieć co zrobić.
44. Jako członek rodziny, chcę aby aplikacja działała po odświeżeniu strony (SSR + cookie), aby cold load był szybki.
45. Jako członek rodziny, chcę aby aplikacja działała w trybie offline (shell + ostatnio cache'owane zdjęcia), aby nie widzieć "błąd sieci" gdy mam słaby zasięg.

## Implementation Decisions

### Architektura i stack

- **Frontend framework:** TanStack Start (SSR + Router + Query) — wymaganie klienta (existing codebase).
- **Backend framework:** Hono na Cloudflare Workers — wymaganie klienta (existing codebase).
- **Database:** Cloudflare D1 (SQLite managed). Wybrany dla maksymalnej prostoty self-hostingu — jedna komenda `wrangler d1 create` i gotowe, zero external dependencies.
- **Image storage:** Cloudflare Images (paid, ~$5/mies per instance). Wybrany dla automatycznej konwersji HEIC, generacji wariantów (thumbnail/full/avif) i braku limitu CPU time w Workerach.
- **PWA:** `vite-plugin-pwa` z Workbox — standardowe wsparcie PWA dla Vite/TanStack Start.
- **Push:** VAPID Web Push protocol, subskrypcje w D1. Wysyłka z Workera przez `fetch` + Web Crypto API (sygnowanie JWT).
- **Styling:** Tailwind v4 + Shadcn (existing).
- **Language:** TypeScript strict, Polish UI (no i18n infrastructure).
- **License:** AGPL-3.0-or-later.

### Boundaries i deep modules

System jest podzielony na 7 deep modules, każdy z wąskim publicznym interfejsem:

1. **Identity** — magic link authentication, session management, member CRUD, revoke flow. Hides cookie lifecycle, token hashing, DB queries.
2. **Instance Bootstrap** — claim-instance flow, setup guard, family config. Hides empty-DB detection, one-shot enforcement.
3. **Image Storage** — abstrakcja nad Cloudflare Images API. Interface: create direct upload, get variant URLs, delete. Hides API details, HEIC handling.
4. **Posts** — CRUD postów z soft delete, cursor pagination dla feedu, edit description.
5. **Comments** — CRUD komentarzy z soft delete, listing per post.
6. **Push Notifications** — subscribe/unsubscribe device, fan-out notify, 410 cleanup. Hides VAPID signing, JWK keys, subscription storage.
7. **Authorization** — rule engine: kto może edytować/usuwać co. Admin override wszędzie.

Moduły komunikują się przez funkcje, nie eventy. Push notifications wywoływane synchronicznie z Posts/Comments po commit (fire-and-forget via `waitUntil`).

### Auth model (detale)

- Token generowany jako 32 bajty crypto-random, base64url-encoded.
- W DB przechowywany jako SHA-256 hash (nie plaintext).
- Magic link format: `https://wspolniak.com/app/u/<token>`.
- Kliknięcie linku → backend weryfikuje token → ustawia HTTP-only, Secure, SameSite=Lax cookie z sesją → redirect do `/app`.
- Sesja = podpisany JWT w cookie, 1 rok ważności (rotating refresh nie jest potrzebny dla tego use case).
- Revoke = zmiana `token_hash` w DB → stary token nie matchuje → stara sesja cookie nadal działa do wygaśnięcia (trade-off: prostsze; admin powinien ostrzec przed revoke). Full revoke sesji = post-MVP.

### Provisioning flow

1. Self-hoster klika "Deploy to Cloudflare" → fork repo → CF tworzy Worker, D1, CF Images binding.
2. Po deploy, `wspolniak.example.workers.dev` (lub custom domain) zwraca landing page.
3. Wejście na `/setup` → form (family_name, admin_name) → backend sprawdza `instance_config.setup_completed` → jeśli false: tworzy user rolą `admin`, generuje magic link, oznacza `setup_completed=true` → wyświetla link adminowi.
4. Dalsze wejścia na `/setup` zwracają 404.

### URL routing

- `wspolniak.com/` — landing page (publiczny, SSR, statyczny content z opisem projektu + GitHub link + "Deploy to Cloudflare" button).
- `wspolniak.com/setup` — claim instance (404 jeśli setup completed).
- `wspolniak.com/app/u/:token` — magic link handler (ustawia cookie, redirect do `/app`).
- `wspolniak.com/app` — feed (wymaga cookie).
- `wspolniak.com/app/new` — nowy post (wymaga cookie).
- `wspolniak.com/app/post/:id` — single post + komentarze.
- `wspolniak.com/app/admin` — panel admina (wymaga role=admin).
- `wspolniak.com/api/*` — Hono API.

### Upload flow (detale)

1. Klient otwiera formularz nowego posta, wybiera 1-10 plików.
2. Klient woła `POST /api/images/upload-url` dla każdego pliku → backend woła Cloudflare Images API `/v1/direct_upload` → zwraca `uploadURL` + `cf_image_id`.
3. Klient uploaduje plik bezpośrednio do `uploadURL` (bypass Worker, oszczędza CPU time).
4. Po zakończeniu wszystkich uploadów: klient woła `POST /api/posts` z listą `cf_image_id` + description.
5. Backend weryfikuje że każdy `cf_image_id` istnieje w CF Images (GET check), zapisuje post + post_images w D1, triggeruje push fan-out.

### Push fan-out

1. Po utworzeniu posta: pobrać wszystkich members (oprócz autora) z aktywnymi push_subscriptions, wysłać push do każdego przez `waitUntil(fetch(endpoint, { body: encrypted, headers: { Authorization: VAPID JWT } }))`.
2. Po utworzeniu komentarza: pobrać author_id posta, pobrać jego subskrypcje, wysłać (jeśli autor komentarza != autor posta).
3. Jeśli push endpoint zwróci 410 Gone → usunąć subskrypcję z D1.
4. Każda push notification zawiera: title, body, icon, URL do deep-linkowania do posta.

### Limity i enforcement

| Limit | Wartość | Enforcement |
|---|---|---|
| Zdjęć per post | 10 | Client-side + backend validation |
| Rozmiar zdjęcia | 15 MB | Cloudflare Images direct upload size limit config |
| Postów dziennie per user | 50 | D1 count query przy `POST /api/posts` |
| Długość description | 2000 znaków | Zod schema |
| Długość komentarza | 1000 znaków | Zod schema |

Brak rate limitingu poziomu HTTP w MVP (happy path assumption — admin nie spamuje rodziny).

### Offline PWA strategy

- **App shell** (HTML, JS, CSS, Shadcn komponenty) — `CacheFirst` z revalidation on update.
- **Cloudflare Images URLs** (immutable per `cf_image_id`) — `CacheFirst`.
- **API feed (`GET /api/posts`)** — `NetworkFirst` z fallback cache (ostatni załadowany feed dostępny offline).
- **API mutations (POST/PUT/DELETE)** — tylko online, wyświetlić "Brak połączenia" jeśli offline.

## Validation Strategy

### Per user story

Każda user story ma acceptance criteria weryfikowane przez kombinację:

- **Unit tests** (Vitest, bez DOM) dla deep modules na granicy publicznego API.
- **Integration tests** (Vitest + in-memory D1 via `@cloudflare/workers-types` miniflare) dla Hono routes i server functions.
- **E2E smoke tests** (manualne) dla PWA installability, push notifications na iOS/Android, magic link flow.

### Deep module completeness

Każdy z 7 deep modules uznaje się za "done" gdy:

1. **Identity** — można utworzyć membera, wygenerować link, zalogować się, revoke'ować, wylogować. Testy pokrywają happy path + invalid token + revoked token.
2. **Instance Bootstrap** — pusta DB → `/setup` działa, po setup → `/setup` zwraca 404. Testy idempotency.
3. **Image Storage** — mock CF Images API, test: create upload URL, get variants, delete. Real integration smoke test manualnie.
4. **Posts** — CRUD, pagination, soft delete, limits enforcement. Testy: autoryzacja (członek nie edytuje cudzego posta), admin override.
5. **Comments** — CRUD, pagination, soft delete. Testy: autoryzacja analogiczna do Posts.
6. **Push Notifications** — mock fetch, test: fan-out logic (kogo powiadamiać), 410 cleanup. Manual smoke test na realnym iPhone 16.4+ PWA i Android.
7. **Authorization** — pure functions, 100% branch coverage unit tests.

### Quality gates przed merge

- `pnpm types` — zero błędów TypeScript
- `pnpm lint` — zero błędów Biome
- `pnpm test` — wszystkie testy zielone
- `pnpm knip` — zero unused exports/files
- Manual: feed ładuje się <2s na 3G (Chrome DevTools throttle)
- Manual: PWA installable score w Lighthouse = 100

### MVP release criteria

Production deploy na `wspolniak.com` uznaje się za gotowy gdy:

1. Autor i min. 3 członków jego rodziny (różne urządzenia: iPhone, Android, desktop) używają aplikacji przez min. 1 tydzień bez krytycznych bugów.
2. Push notifications działają na wszystkich tych urządzeniach.
3. Landing page + README + LICENSE + "Deploy to Cloudflare" button są publiczne.
4. Minimum jedna osoba spoza autora self-hostowała instancję w ramach beta test.

## Out of Scope

### Explicitly post-MVP (planned)

- Reakcje / lajki na postach
- Albumy (grupowanie zdjęć w nazwane kolekcje)
- @mentions w komentarzach
- Wątkowane komentarze (nested replies)
- Edycja zdjęć (crop, rotate, filters)
- EXIF stripping (szczególnie GPS)
- PIN-based device activation
- i18n (tłumaczenia UI)
- Avatary członków rodziny
- Branding instancji (logo, kolory)
- Export danych (user self-service)
- Delete account (user self-service)
- Rate limiting
- Backup cron do R2/external
- Share magic link przez WhatsApp integration
- Admin impersonation (admin loguje się jako członek do debugowania)
- License enforcement (telemetry, keys)
- Reset hasła (nie ma hasła)
- Email notifications
- Stories / ephemeral content

### Explicitly never (not planned)

- Multi-tenancy (jeden worker, wiele rodzin)
- Docker deployment (utrzymanie CF-only spójności)
- Postgres/MySQL support
- Natywne aplikacje mobilne (iOS/Android) — PWA wystarcza
- Publiczne profile / shareable links do postów
- Algorithmic feed (ranking, personalizacja)
- Reklamy
- Platforma SaaS (hosted version for other families)

## Further Notes

### License headers (SPDX)

Po utworzeniu pliku `LICENSE` (AGPL-3.0), każdy nowy source file powinien zawierać na górze:

```
// SPDX-License-Identifier: AGPL-3.0-or-later
```

To jest standard w OSS i ułatwia narzędziom (FOSSA, REUSE) weryfikację compliance.

### Cloudflare Images koszt dla self-hosterów

$5/mies (100k obrazów, 100k delivery) może być barierą dla self-hosterów. **W MVP nie robimy abstrakcji** (YAGNI), ale w README jasno to dokumentujemy jako wymaganie. Jeśli pojawi się uzasadniony request od self-hostera, post-MVP można dodać `ImageStorage` interface z fallback implementacją na R2 bez HEIC conversion.

### RODO / privacy disclaimer

Jako self-hostowany serwis dla "wyłącznie osobistych/domowych celów" (art. 2 ust. 2 lit. c RODO) mieścimy się w exclusion. Jednak inne rodziny używające tego samego kodu mogą nie — dlatego:

- Nie zbieramy telemetry/analytics
- Nie ma persistent identyfikatorów spoza family cookie
- Soft delete pozwala na implementację "right to be forgotten" w post-MVP bez refaktoru
- Landing page powinien mieć disclaimer "For personal family use only"

### Testing strategy — realne urządzenia krytyczne

Symulatory i Chrome DevTools **nie** wykryją niektórych bugów:

- iOS Safari PWA push wymaga fizycznego iPhone 16.4+ z "Dodaj do ekranu głównego"
- HEIC upload flow wymaga realnego iPhone'a robiącego zdjęcie (nie file z dysku)
- Android Chrome PWA installability banner różni się od desktop Chrome

Przed release: przetestować na min. jednym iPhone 16.4+, jednym Android, desktop Chrome i Safari.

### Dependencies między modułami

```
Push Notifications ─┬─> Identity (who to notify)
                    └─> Posts/Comments (trigger source)

Posts ──> Authorization ──> Identity
Comments ──> Authorization ──> Identity
Posts ──> Image Storage
Instance Bootstrap ──> Identity (create first admin)
```

Kolejność implementacji (dla `/carve` vertical slices):
1. Instance Bootstrap + Identity (claim flow + magic link login)
2. Image Storage (direct upload do CF Images)
3. Posts (create, feed, view)
4. Comments
5. Push Notifications
6. Authorization refinement + admin panel
7. PWA finishing touches (offline, install prompt)
